# Decision 0001 — `consumer-conformance-shapes.ttl` is ontology-owned (Option C)

## Context

On 2026-05-04, the 5 consumer-conformance Claim shapes (`M1' SliceCoherenceClaim`, `M3 PracticeCOConsistency`, `M4 MechanismCOConsistency` via CO chain, `M4-card ClaimWellFormedness`, `M4-card ClaimTargetReferentialIntegrity`) — originally shipped 2026-05-03 in commit `f25eb48` as a sibling file at `formal/appsec_core/03-shacl/shapes/consumer-conformance-shapes.ttl` — were **consolidated** into the apparatus shapes file `appsec-core-v0-shapes.ttl` and pinned at `apparatus-shacl-pyshacl-v2` (commit `ee73c19`). The sibling file was removed; the apparatus contained 11 shapes (6 ontology + 5 claim).

On 2026-05-05, during the ACR-004 execution sprint, commit `7ee0373` ran `build_shacl.py`. The build script emits `appsec-core-v0-shapes.ttl` from canonical YAML + entity schema, with no awareness of model invariants from Decision 0003 + Amendment 1. The regen-from-scratch wiped the 5 claim shapes that consolidation had introduced.

Cartographer detected the regression while emitting substrate v6: the substrate v6 grounding pipeline relies on the claim shapes to validate consumer conformance. Cartographer restored the shapes from `ee73c19` to run the gate; substrate v6 itself is valid. The branch `acr004-output-rendering` shapes file at HEAD `c6f6059` was the regression artefact.

## Decision

Programme-lead Pedro Farinha ratified **Option C** on 2026-05-05:

1. **`consumer-conformance-shapes.ttl` is an ontology-owned artefact.** It lives at `formal/appsec_core/03-shacl/shapes/consumer-conformance-shapes.ttl` alongside `appsec-core-v0-shapes.ttl`.
2. **Maintained by Archon.** The 5 Claim shapes are derived from Decision 0003 + Amendment 1 model invariants. Identification provenance (Cartographer/ESI surfacing) is historical; ownership is Archon (the ontology owner).
3. **NOT regenerated by `build_shacl.py`.** The build script regenerates only the 6 schema-derived ontology shapes (Slice / CO / Practice / Mechanism / Artifact / EvidencePattern). Re-running `build_shacl.py` must never modify `consumer-conformance-shapes.ttl`.
4. **Apparatus tags declare composition.** Each `apparatus-shacl-pyshacl-vN` tag declares both files explicitly with their commit anchors. The pyshacl runner (`validate_pyshacl.py`) loads both files as a merged shapes graph before validation.
5. **Single SHACL concern preserved.** Both files validate the same ontology; the split is operational (schema-derived regenerable vs invariant-derived hand-maintained), not semantic.

## Why not Option A (consolidated single file)

Option A — keep both shape sets in one consolidated `appsec-core-v0-shapes.ttl` — was the apparatus-v2 design. It failed because the file is overloaded with two regen lifecycles in conflict:

- Schema-derived ontology shapes regenerate naturally when YAML changes (build_shacl.py is the safe path).
- Invariant-derived claim shapes change only when Decision 0003 receives an amendment (cross-persona dispatcher path).

A single file forces every YAML change to route through manual claim-shape preservation. Discipline-only enforcement (verify-first reading before each `build_shacl.py` run) failed in `7ee0373`. Option C makes the boundary structural, not behavioural.

## Why not Option B (build_shacl.py learns about claim shapes)

Option B — extend `build_shacl.py` to emit the 5 claim shapes from a separate input source — confuses two responsibilities:

- `build_shacl.py` is YAML-schema → SHACL. Adding model-invariant logic mixes two encodings (YAML schema + Decision 0003 invariants) into one generator.
- The model invariants are not derivable from the YAML schema; they are a separate authority surface.

Option C keeps each file with one source of truth and one regen pathway.
