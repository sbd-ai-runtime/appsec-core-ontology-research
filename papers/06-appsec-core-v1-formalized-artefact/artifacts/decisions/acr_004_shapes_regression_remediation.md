## Why this happened (root cause)

`build_shacl.py` emits 6 ontology shapes from canonical YAML schema. It has no awareness of the model invariants from Decision 0003 + Amendment 1. Re-running it from scratch wipes any consolidated content not derivable from schema. Discipline-only enforcement (verify-first read of the existing file before regen) failed in `7ee0373` — the runtime equivalent of "I forgot the file already had something I needed to preserve."

The structural fix is to put the two concerns (schema-derived ontology shapes vs invariant-derived claim shapes) in two files with two regen lifecycles, instead of relying on an operator to remember to preserve consolidation in a single file.

## Architectural commitment (Option C, programme-lead 2026-05-05)

Recorded in detail at `agentic/decisions/0001-consumer-conformance-shapes-ontology-owned.md`. Summary:

1. `consumer-conformance-shapes.ttl` is ontology-owned and Archon-maintained.
2. Lives at `formal/appsec_core/03-shacl/shapes/` alongside `appsec-core-v0-shapes.ttl` as a separate file.
3. NOT regenerated by `build_shacl.py` (regen-safety boundary documented in build_shacl.py docstring).
4. Apparatus tags (`apparatus-shacl-pyshacl-vN`) declare composition of both files explicitly.
5. Pyshacl runner (`validate_pyshacl.py`) loads both files as merged shapes graph before validation.
6. Cross-persona update protocol: Decision 0003 amendment → Cartographer dispatches Archon → Archon updates the file in dedicated commit → apparatus tag refresh under programme-lead authority.
